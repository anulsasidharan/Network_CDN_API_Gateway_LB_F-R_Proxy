Load balancer

Distributes incoming traffic across multiple backend servers — the single entry point for your server pool.

L4 / L7 What it does
Prevents any single server from becoming a bottleneck. Sits in front of your server pool and routes each request to a healthy backend.
Algorithms
Round robin — requests sent to each server in turn. Simple, no state.
Least connections — route to the server with fewest active connections.
IP hash — same client IP always hits the same server (session stickiness).
Weighted — high-spec servers receive a larger traffic share.
L4 vs L7
L4 (Transport) — routes by IP + port. Fast, but cannot inspect HTTP headers.
L7 (Application) — routes by URL, headers, cookies. Smarter — can send /api/* to one pool and /static/* to another.
Health checks
The load balancer periodically pings each server. If a server fails to respond, traffic stops going to it automatically. Once recovered, it is re-added to the pool — the core of high availability.
Reverse proxy

Sits in front of your servers and accepts requests on their behalf — clients never see the real server addresses.

Inbound What it does
Enables SSL termination, compression, caching, and request routing. Clients talk to the proxy; the proxy talks to your backends.

Key capabilities
SSL termination — decrypt HTTPS once here, not on every backend
Request buffering — absorb slow clients before hitting origin
Response caching — serve repeated reads without touching backend
Compression — gzip/brotli responses to reduce bandwidth
Path-based routing — /api → app server, /static → file server
vs Forward proxy
Forward proxy — client knows about it. Hides the client from servers.
Reverse proxy — client doesn't know about it. Hides servers from clients.
Forward proxy

Sits between clients and the internet — the destination server sees the proxy's IP, not the client's.

Outbound What it does
Clients are configured to send requests through it. Used for access control, anonymity, monitoring, and caching outbound traffic.

Use cases
Corporate network filtering — block social media, log traffic
Anonymisation — mask user IPs (VPNs use this concept)
Geo-bypass — request appears to come from the proxy's country
Caching repeated outbound requests (e.g. package registries)
Real examples

VPN services — forward proxy at the OS network level
API gateway

A specialised reverse proxy for APIs — the front door to your microservices.

Cross-cutting What it does
Handles authentication, rate limiting, request transformation, routing, and observability in one place — so every service doesn't have to implement these individually.
Auth & security
JWT validation, OAuth2 token introspection, API key management, IP allowlisting, WAF integration
Traffic control
Rate limiting per client/IP, quota enforcement, circuit breaking, retry logic, canary routing
Transformation
Request/response header rewriting, protocol translation (REST↔gRPC), request aggregation, versioning

Content delivery network (CDN)

A globally distributed network of edge servers that cache content close to users — reducing latency and absorbing traffic spikes.

Edge caching What it does
PoPs (Points of Presence) serve cached content from the nearest edge node instead of your origin server. Critical for video, images, and static assets at scale.
What gets cached
Static assets — JS, CSS, images, fonts, videos
API responses with Cache-Control headers
HTML pages (with caution — can't cache per-user)
Dynamic content — usually bypassed via cache-busting rules
Beyond caching
DDoS mitigation at the edge (Cloudflare, Akamai)
Edge compute — run logic (auth, redirects) at the PoP
Image optimisation — resize/convert on the fly
TLS termination at the edge — faster handshakes

Nginx — practical configuration

Real Nginx configs demonstrating reverse proxy, traffic routing, and load balancing algorithms.

Reverse proxy with upstream pool
upstream backend_pool {
  server 10.0.0.1:8080;
  server 10.0.0.2:8080;
  server 10.0.0.3:8080;
  keepalive 32;
}

server {
  listen 443 ssl http2;
  server_name api.example.com;

  # SSL termination
  ssl_certificate /etc/ssl/certs/cert.pem;
  ssl_certificate_key /etc/ssl/private/key.pem;

  location / {
    proxy_pass http://backend_pool;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
  }
}
Traffic routing — path-based
location /api/ {
  proxy_pass http://api_servers;
}

location /static/ {
  root /var/www;
  expires 30d;
  add_header Cache-Control "public, immutable";
}

location /ws/ {
  proxy_pass http://ws_servers;
  proxy_http_version 1.1;
  proxy_set_header Upgrade $http_upgrade;
  proxy_set_header Connection "upgrade";
}
Load balancing algorithms in Nginx
# Round robin (default — no directive needed)
upstream rr_pool { server s1:8080; server s2:8080; }

# Least connections
upstream lc_pool { least_conn; server s1:8080; server s2:8080; }

# IP hash (sticky sessions)
upstream sticky_pool { ip_hash; server s1:8080; server s2:8080; }

# Weighted
upstream wt_pool {
  server s1:8080 weight=3;
  server s2:8080 weight=1;
}
Real-world architecture examples

How Netflix, YouTube, and Amazon use each networking component at global scale.

Netflix
CDN: Open Connect — ISP-embedded appliances serving ~99% of traffic
API Gateway: Zuul — auth, rate limiting, A/B routing
Load balancer: Ribbon + Eureka (client-side) across AWS regions
Reverse proxy: Envoy sidecar per microservice (service mesh)
YouTube
CDN: Google PoPs in 200+ countries; video segments cached at edge
API Gateway: Cloud Endpoints + Apigee (YouTube Data API v3)
Load balancer: Google Cloud LB — anycast IP to nearest datacenter
Reverse proxy: GFE (Google Front End) terminates SSL
Amazon
CDN: CloudFront — 450+ PoPs for images, JS, CSS
API Gateway: AWS API Gateway (Marketplace API)
Load balancer: ALB (L7) + NLB (L4) for internal traffic
Forward proxy: Squid for internal developer traffic control
